# Markup data model (highlighting, annotations, comments)

Design sketch for issues [#1 (highlighting)](https://github.com/jnahian/reader.md/issues/1),
[#2 (annotations)](https://github.com/jnahian/reader.md/issues/2), and
[#3 (comments with resolve)](https://github.com/jnahian/reader.md/issues/3).

The core move: **stop treating highlight / annotation / comment-thread as three
types — they are one anchored entity with optional parts filled in.**

## One entity

```swift
// A single anchored mark. Highlight, annotation, and comment thread are all
// THIS — they differ only in how many `comments` it has and `resolved`.
struct Mark: Codable, Identifiable {
    let id: UUID
    var anchor: TextAnchor
    var color: HighlightColor      // every mark renders as a colored range
    var comments: [Comment]        // []=highlight · 1=annotation · 1+=thread
    var resolved: Bool             // only meaningful once comments is non-empty
    let createdAt: Date
    var updatedAt: Date
}

struct Comment: Codable, Identifiable {
    let id: UUID
    var author: String             // local display name — no accounts, app is offline
    var text: String
    let createdAt: Date
    var editedAt: Date?
}

enum HighlightColor: String, Codable, CaseIterable {
    case yellow, green, blue, pink, purple
}
```

## The anchor (the hard part, shared by all three)

```swift
// Robust anchor over the *rendered* text (what the user selects), NOT DOM
// offsets — the HTML is regenerated by marked on every reload. Re-resolved in
// bridge.js after each render. Modeled on the W3C TextQuoteSelector.
struct TextAnchor: Codable {
    var quote: String      // the exact selected text
    var prefix: String     // ~32 chars before it — disambiguates repeated quotes
    var suffix: String     // ~32 chars after it
    var startOffset: Int   // fallback tiebreaker: char index into the plain-text render
}
```

Resolution order in JS: find `quote` occurrences in the rendered `textContent` →
pick the one whose surrounding text matches `prefix`/`suffix` → if still
ambiguous, nearest `startOffset` → if `quote` is gone entirely, the mark is
**orphaned** (runtime state, surfaced in a list, never silently dropped).

## Storage

```swift
// One JSON file per viewed document, in Application Support.
struct MarkDocument: Codable {
    var schemaVersion: Int   // 1 — lets the format evolve
    var filePath: String     // absolute path when last seen
    var contentHash: String  // sha256 of file at last successful resolve
    var marks: [Mark]
}
```

`~/Library/Application Support/Reader.md/marks/<sha256(absolutePath)>.json`

**Keying tradeoff** (the one real decision): filename derived from the **path**
→ survives content edits, lost on rename/move. `contentHash` stored *inside* →
lets you detect "file changed since marks were made" and re-resolve/flag orphans.
Content-based keying would survive moves but break on every edit. Path-key +
stored hash is the least-surprising v1; rename-tracking is a later enhancement,
not a blocker.

## How the three features collapse onto it

| Feature (#) | `color` | `comments` | `resolved` | UI |
|---|---|---|---|---|
| **Highlight** (#1) | set | `[]` | — | colored range |
| **Annotation** (#2) | set | `[oneComment]` | ignored | range + margin note icon |
| **Comments** (#3) | set | `[c1, c2, …]` | toggled | range + thread card, resolve/reopen |

The UI branches on `comments.count` and `resolved` — no type switch, no second
storage system. A highlight *becomes* an annotation when you add the first
comment; an annotation *becomes* a thread when you add the second. Same row,
growing.

## Bridge wiring (fits the existing token-bump pattern)

- **Swift → JS**, after every render: `window.ReaderMd.applyMarks(json)` — JS
  wraps resolved ranges in `<mark data-mark-id>` spans + thread indicators.
- **JS → Swift** message handlers (new): `createMark` (JS sends computed anchor +
  selection), `addComment`, `resolveMark`, `deleteMark`. Swift mutates the
  `MarkDocument`, persists, re-applies.
- Re-apply is just another consumer of the reload cycle already in place.

**The payoff of unifying:** resolve, orphan-handling, persistence, and anchoring
are written **once**. Build #1 as `Mark` with empty `comments`, and #2/#3 are
mostly UI on top of the same store — no migration between two models later.
